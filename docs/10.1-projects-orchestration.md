# 14 - Projects Orchestration

Managed-mode feature enabling GoClaw to orchestrate source code projects on the same machine where GoClaw runs. Sessions use Claude Code CLI as the execution engine for interactive coding tasks.

---

## 1. Overview

**Project** is the top-level entity representing a source code repository/directory managed by GoClaw. Each project points to a working directory on the host machine and can have multiple sessions.

**Session** is a Claude Code CLI process spawned within a project's working directory. Sessions receive prompts, stream real-time output, and persist all events to PostgreSQL.

Key capabilities:
- Create and configure projects (working directory, allowed tools, environment, model config)
- Start interactive sessions with real-time streamed output via WebSocket
- Send follow-up prompts to running sessions (resume flow)
- Monitor session metrics (tokens, cost, execution time)
- Access full event logs for audit and replay

Future: Projects will also host MCP server configs, custom tool sets, and other per-project features beyond Claude Code sessions.

---

## 2. Architecture

```
Web UI (/projects, /projects/sessions/:id)
    ↓
HTTP API (/v1/projects/*, /v1/project-sessions/*) + WebSocket RPC (projects.*)
    ↓
ProjectsHandler + ProjectsMethods
    ↓
ProcessManager (spawn, monitor, lifecycle)
    ↓
Claude Code CLI (child process, --print --output-format stream-json)
    ↓ (NDJSON stdout)
StreamParser
    ↓
ProjectStore (PostgreSQL)
    ↓
Message Bus (broadcast project.output, project.session.status events)
    ↓
WebSocket Clients (real-time updates)
```

### Concurrency Model

- **Per-project session limit** via `max_sessions` (default 3)
- **Thread-safe process map** (ProcessManager.mu guards process registry)
- **Context cancellation** propagates kill signal to child process on shutdown

## 4. Access Control

Projects enforce a 4-level access check priority:

```
1. System owner (isOwner from gateway config) → full access
2. Project owner (project.owner_id == userID) → full access
3. Explicit member (project_members table) → read access
4. Team-linked access:
   - project.team_id → team.created_by matches userID → read access
   - project.team_id → team.settings.allow_user_ids contains userID → read access
```

**Modify operations** (update, delete, add/remove members) are restricted to project owner + system owner.

Both HTTP API (`internal/http/projects.go`) and WebSocket RPC (`internal/gateway/methods/projects.go`) enforce these checks via `canAccess()` and `canModify()` helpers.

### ListAccessibleProjects

For non-system-owner users, `ListAccessibleProjects` returns the UNION of:
- Projects owned by the user (`owner_id = userID`)
- Projects where user is an explicit member (`project_members`)
- Projects linked to teams created by the user (`agent_teams.created_by`)

Team `allow_user_ids` check is performed in Go (JSONB settings field is complex to query in SQL).

## 5. HTTP API

Handlers in `internal/http/projects.go`. Auth via Bearer token + `X-GoClaw-User-Id` header. All endpoints enforce access control (see Section 4).

### Projects

| Method | Path | Purpose | Access |
|--------|------|---------|--------|
| GET | `/v1/projects` | List accessible projects | Filtered by access |
| POST | `/v1/projects` | Create new project | Any authenticated user |
| GET | `/v1/projects/{id}` | Get project details | canAccess |
| PUT | `/v1/projects/{id}` | Update project config | canModify (owner only) |
| DELETE | `/v1/projects/{id}` | Delete project (cascades) | canModify (owner only) |
| GET | `/v1/projects/{id}/sessions` | List project sessions | canAccess |
| POST | `/v1/projects/{id}/sessions` | Start new session | canAccess |

### Sessions

| Method | Path | Purpose | Access |
|--------|------|---------|--------|
| GET | `/v1/project-sessions/{id}` | Get session status + metrics | canAccess (parent project) |
| PATCH | `/v1/project-sessions/{id}` | Update session (label) | Any authenticated |
| DELETE | `/v1/project-sessions/{id}` | Delete session | Any authenticated |
| POST | `/v1/project-sessions/{id}/prompt` | Send prompt to running session | canAccess (parent project) |
| POST | `/v1/project-sessions/{id}/stop` | Stop running session | canAccess (parent project) |
| GET | `/v1/project-sessions/{id}/logs` | Get session event logs | canAccess (parent project) |

### Members

| Method | Path | Purpose | Access |
|--------|------|---------|--------|
| GET | `/v1/projects/{id}/members` | List project members | canAccess |
| POST | `/v1/projects/{id}/members` | Add member (`{user_id, role}`) | canModify (owner only) |
| DELETE | `/v1/projects/{id}/members/{user_id}` | Remove member | canModify (owner only) |

> **Note:** Session endpoints use `/v1/project-sessions/` (not `/v1/projects/sessions/`) to avoid Go ServeMux route conflict with `/v1/projects/{id}/sessions`.

## 8. ProcessManager Lifecycle

`internal/claudecode/manager.go` manages Claude Code CLI processes.

### State Transitions

```
STARTING → RUNNING → COMPLETED | FAILED | STOPPED
```

### Start Flow

1. Check active session count vs `max_sessions`
2. Create session record (status=starting)
3. Build CLI args (`claude --print --output-format stream-json -p "prompt"`)
4. Spawn process, capture stdout/stderr
5. Update status to running + store PID
6. Parse NDJSON stream events → persist logs + broadcast events
7. On process exit → update status, metrics, timestamps

### Shutdown

On graceful shutdown: SIGTERM → wait → SIGKILL if needed → release worktrees.
